Background

Twist Commands

In ROS 2, the standard way to tell a robot how to move is by publishing a geometry_msgs/msg/Twist message. A Twist message represents velocity in free space and contains two main components:

  • Linear Velocity: Movement along the axes (x, y, z) measured in m/s.
  • Angular Velocity: Rotation around the axes (x, y, z) measured in rad/s.

Because the Romi is a ground-based robot that cannot fly or strafe, we only care about two of these values: linear velocity moving forward/backward (linear.x) and angular velocity rotating left/right (angular.z).

Differential Drive Kinematics

Your robot has two independently driven wheels. To make the robot move at a specific linear velocity (v) and angular velocity (ω), we must calculate the required speed for the left wheel (vL) and the right wheel (vR). This mathematical translation is called inverse kinematics.

For a differential drive robot, the equations are:

  • Left wheel velocity: vL = v - (ωL / 2)
  • Right wheel velocity: vR = v + (ωL / 2)

Where:

  • v is the desired linear velocity (m/s)
  • ω is the desired angular velocity (rad/s)
  • L is the wheelbase, which is the distance between the two wheels (m)

PID Control Basics

Once we calculate the desired vL and vR, we need to make sure the physical motors actually spin at those speeds. Due to battery voltage drops, friction, and varying terrain, sending a static power value to the motors will result in inconsistent speeds. We solve this using a closed-loop controller.

A PID controller calculates an error value e(t) as the difference between a desired setpoint (target velocity) and a measured process variable (actual velocity calculated from encoder counts). It then applies a correction based on three terms:

u(t) = Kpe(t) + Ki∫e(t)dt + Kd(de(t)/dt)
  • Proportional (Kp): Reacts to the present error. A larger error results in a larger immediate push.
  • Integral (Ki): Reacts to the accumulation of past errors. If the robot is slightly too slow for a long time, the integral term builds up to give it the extra nudge it needs to reach the target.
  • Derivative (Kd): Predicts future error based on the rate of change.

Note on implementation: In practice, the derivative term is highly susceptible to sensor noise from the encoders, which can cause erratic twitching. Therefore, the PI Controller (omitting the D term by setting Kd=0) is the most common and robust implementation for velocity tracking. It should also be noted that, while we will be implementing PI control in a ROS 2 node, such controllers are typically implemented on the low-level driver (the Romi32U4 in our case) due to the time-critical nature of control feedback loops. The choice to implement PI control in ROS 2 for this exercise was made to maintain educational clarity and workflow.

Safety and Professional Responsibility: The Risks of Control

As a robotics engineer, your code is the bridge between mathematical abstraction and physical action. In a lab setting, a poorly tuned PID loop might just result in a robot bumping into a table or spinning in circles. However, in industrial or public-facing applications, the stakes are significantly higher.

  • Public Safety and Physical Harm: Inadequate control design in autonomous vehicles, delivery drones, or surgical robots can lead to catastrophic failure.
  • Instability: A divergent PID loop, where the gains are too high, causes oscillations that grow in magnitude. On a Romi, this looks like shaking; on a full-size autonomous vehicle, this could mean a violent swerve.
  • Latency: If your control loop is too slow or sluggish, the robot cannot react to an emergency stop in time.
  • Anti-Windup: In the provided starter code, there is a block of code for integral anti-windup. Without this, if a robot’s wheel is physically stuck, the integral term will grow while trying to correct the error. Once the robot is freed, it can release that stored error as a sudden spike in motor output.

2. Create the Automated Square Test Node

After your velocity controller is working with keyboard teleoperation, create a separate ROS 2 Python node that automatically sends velocity commands to make your robot drive in a 2 ft x 2 ft square.

This test node will publish geometry_msgs/msg/Twist messages to the /cmd_vel topic. The robot should drive forward one foot, turn 90 degrees, and repeat this process four times.

Create the file in the existing romi_base Python package at this exact location:

~/ros2_ws/src/ROMI-ROS2/romi_base/romi_base/square_test_node.py

Copy and paste the following code into square_test_node.py:

#!/usr/bin/env python3

import time

import rclpy
from rclpy.node import Node
from geometry_msgs.msg import Twist


class SquareTestNode(Node):
    def __init__(self):
        super().__init__('square_test_node')

        self.cmd_pub = self.create_publisher(Twist, '/cmd_vel', 10)

        # Commanded robot velocities
        self.forward_speed = 0.10      # meters per second
        self.turn_speed = 0.75         # radians per second

        # Desired square dimensions
        self.side_length = 0.6096      # meters, equal to 2 ft
        self.turn_angle = 1.5708       # radians, equal to 90 degrees

        # Motion durations are calculated from commanded velocities.
        # Tune the velocity controller, not these timing calculations.
        self.forward_time = self.side_length / self.forward_speed
        self.turn_time = self.turn_angle / self.turn_speed

        self.get_logger().info('Square test node started.')
        self.get_logger().info(f'Forward time: {self.forward_time:.2f} seconds')
        self.get_logger().info(f'Turn time: {self.turn_time:.2f} seconds')

        time.sleep(2.0)

        self.drive_square()
        self.stop_robot()

        self.get_logger().info('Square test complete.')

    def publish_cmd_for_duration(self, linear_x, angular_z, duration):
        msg = Twist()
        msg.linear.x = linear_x
        msg.angular.z = angular_z

        start_time = time.time()

        while time.time() - start_time < duration:
            self.cmd_pub.publish(msg)
            time.sleep(0.05)

        self.stop_robot()
        time.sleep(0.5)

    def stop_robot(self):
        msg = Twist()
        msg.linear.x = 0.0
        msg.angular.z = 0.0

        for _ in range(10):
            self.cmd_pub.publish(msg)
            time.sleep(0.05)

    def drive_square(self):
        for side in range(4):
            self.get_logger().info(f'Driving side {side + 1}')
            self.publish_cmd_for_duration(self.forward_speed, 0.0, self.forward_time)

            self.get_logger().info(f'Turning corner {side + 1}')
            self.publish_cmd_for_duration(0.0, self.turn_speed, self.turn_time)


def main(args=None):
    rclpy.init(args=args)
    node = SquareTestNode()
    node.destroy_node()
    rclpy.shutdown()


if __name__ == '__main__':
    main()

6. Tuning

With the robot safely on the ground, use the keys indicated by teleop_twist_keyboard to send velocity commands (Twist messages).

  • If your robot behaves erratically, is unresponsive, or oscillates violently, you will need to tune your Kp and Ki gains.
  • Get started by uncommenting the Dashboard Logging code underneath the code where we publish setpoints to the motors. Use the logs printed by base_controller_node to help you tune your gains.
  • Tip: Start with Ki at 0. Increase Kp until the wheels respond snappily to your commands but do not oscillate. Then, slowly increase Ki until the wheels accurately reach and hold the target speed under physical resistance.

Important: The square test is intended to evaluate whether your velocity controller causes the robot to match commanded Twist velocities. The forward and turn durations are calculated from the commanded velocity values. If your robot does not complete the square accurately, do not change the timing values in the square test node to hide the problem. Instead, tune your velocity controller gains and verify that the measured wheel speeds match the commanded wheel speeds.
