Background

System Architecture

In the diagram pictured above:

  • The Raspberry Pi 4B acts as the "Brain." It handles high-level intelligence—such as processing LiDAR data or calculating motion plans—using the robust ROS 2 ecosystem.
  • The Romi 32U4 Control Board acts as the "Nervous System." It is a dedicated microcontroller responsible only for low-level execution, such as generating PWM signals for motors and counting raw encoder pulses.
  • Your Windows Laptop acts as the "Mission Control." It provides a remote console to launch software, visualize sensor data, and send manual override commands.

This is a classic robotics paradigm where the "thinking" is separate from the "doing." By letting a low-level controller handle the fast, repetitive task of spinning motors, the main computer stays free to focus on computationally heavy tasks like navigation and obstacle avoidance.

ROS2 Basics

ROS2 (Robot Operating System) is not actually an operating system like Windows or Linux, but can be thought of as a sophisticated communication framework. Its primary job is to provide a standardized way for different software programs, potentially running on different computers, to talk to one another. In the context of the Romi robot, ROS2 allows your Windows laptop and a Raspberry Pi to work together as a single cohesive unit. Instead of writing complex networking code to send data over Wi-Fi, ROS 2 handles the "plumbing" in the background, allowing you to focus on the logic of your robotics applications.

The fundamental building blocks of this communication are Nodes and Topics. A Node is a small, specialized program designed to perform one specific task—such as reading encoder counts, blinking an LED, or processing sensor data. For these nodes to share information, they use Topics, which act like dedicated "bus routes" or radio channels. One node "Publishes" data to a topic, while any other node interested in that data "Subscribes" to it. For example, in this lab, your laptop will run a node that publishes a command to a topic, and the Raspberry Pi will subscribe to that same topic to trigger a physical action on the Romi. This "Publisher-Subscriber" model is what makes ROS 2 incredibly flexible; you can add or remove nodes without ever having to rewrite the entire system.

In this exercise, we will put this Publisher-Subscriber model into practice to verify your cross-device toolchain. You will launch a setup logic node on your Windows laptop that acts as the high-level controller, publishing commands to a dedicated topic called /leds. Simultaneously, the Raspberry Pi will run a hardware bridge node that subscribes to this topic, listening for those messages and translating them into low-level hardware signals. By successfully observing the physical LEDs on the Romi board toggle in sync with the output on your laptop terminal, you will confirm that your entire communication pipeline is properly configured and fully operational.

Procedure

To set up the robot and toolchain, we will follow the steps below:

  1. Assemble the robot kit
  2. Install Arduino IDE to flash a program to the Romi32U4 Control Board
  3. Flash a “golden image” to your SD card for use in Raspberry Pi
  4. Install ROS2 on your Windows laptop in Windows Subsystem for Linux (WSL)
  5. Install and configure VS Code for code development
  6. Validate basic inter-device communication

Step 1.0: Assemble the Robot

IMPORTANT: Step 1 is only needed for those who obtained their own personal Romi32U4 robot kits. If you are using the provided robots in the lab, you may proceed to Step 2.

If you are assembling your own robot at home, check out the assembly guide first! If you are using one of the robots provided on campus, you can skip the assembly step.

Step 1.1: Install Arduino IDE

  • Go to Arduino’s website to download Arduino IDE.
  • Select the version that corresponds to your operating system.
  • Once downloaded, run the installer. On Windows, the file should be named something like “arduino-ide_<version number>_Windows_64bit.exe”.
  • Once run, the installation wizard will take you through several steps to install Arduino IDE.
  • After you click “Finish” and launch Arduino IDE, you may be prompted to install a variety of drivers and grant device permissions. Most of these are not required for this course, but if you are comfortable, it is safest to greenlight everything.

Step 1.2: Set Up Arduino IDE for Romi32U4

Install Windows Drivers

Before connecting a Pololu 32U4 board to your Windows machine, install the drivers first.

  1. Download the A-Star Windows Drivers and extract the zip to a temporary folder.
  2. Open the “a-star-windows” folder.
  3. Right-click on “a-star.inf” and click “Install.”

Add Pololu A-Star Board Support

To enable support for the Pololu Romi32U4, you will need to add Pololu’s board package URL to the Arduino IDE using the Boards Manager.

  1. Go to File > Preferences > Settings.
  2. Find “Additional boards manager URLs.”
  3. Copy and paste the following URL:
    https://files.pololu.com/arduino/package_pololu_index.json
  4. Click “OK”.
  5. In Tools > Board > Boards Manager, search for “Pololu A-Star Boards”.
  6. Select the latest version and click “Install”.

You should now be able to select your board by clicking “Select Board” and searching for “Pololu A-Star 32U4”.

Install the Romi32U4 Library

Finally, you should install the Romi32U4 library, which helps programmers easily interface with all of the on-board hardware on the control board.

  1. Go to Sketch > Include Library > Manage Libraries.
  2. Search for “Romi32U4”.
  3. Select the latest version and click “Install”.

Step 1.3: Program the Romi

We now need to flash the RomiRPiSlaveDemo.ino program to your Romi32U4. The program enables communication between the Romi32U4 and a Raspberry Pi via a shared I2C register. With this shared register in place, the Romi32U4 becomes a hardware interface that simply acts based on values in the register, while the Raspberry Pi is the one sending the high-level commands.

Download the RomiRPiSlaveDemo.ino, and open it in Arduino IDE. Click the “Upload” button to flash the program to the Romi32U4 Control Board.

Step 2: Raspberry Pi Setup

Now that the Romi is ready to talk to a Raspberry Pi, let’s set one up. This process has been made as simple as possible for you:

  1. Download the Class Pi Image. This .img.xz file contains a preconfigured Ubuntu 22.04 operating system with all the dependencies you will need for the class already installed and system-level configurations already made.
  2. Download and install Raspberry Pi imager.
  3. Mount your SD card in Windows.
  4. In Raspberry Pi Imager:
    • Select “Raspberry Pi 4” as device.
    • Under OS selection, scroll to the bottom and select “Use Custom.”
    • Find and select the .img.xz file you downloaded in Step 1.
    • Under Storage selection, select your SD card.
    • Flash the image.

Your SD card is now ready to be inserted into a Raspberry Pi 4B (we will do this later).

Step 3: ROS 2 in WSL2

ROS 2 workflow is the smoothest using Linux, so we will be using Windows Subsystem for Linux 2 (WSL2)—an extremely lightweight Linux VM—to host ROS 2.

  1. Open PowerShell as an administrator.
  2. Run wsl --install
  3. Run wsl --install -d Ubuntu-22.04
  4. Ubuntu should launch after install → Set up your user/password as prompted.
  5. After setting up user, run sudo apt update && sudo apt install -y
  6. Follow the steps here to install ROS 2 (make sure to install the desktop version and the development tools).

Once ROS 2 is installed in WSL2, you will want to add the setup script to your .bashrc to avoid having to source it every time you boot up WSL. Do this by running:

echo "source /opt/ros/humble/setup.bash" >> ~/.bashrc

We now need to ensure that multicast network communication can work properly:

  1. Open the WSL Settings GUI (press Windows key and search for this).
  2. Navigate to the Networking settings, and ensure that "Networking mode" is set to "Mirrored".
  3. Open PowerShell as an administrator.
  4. Run Set-NetFirewallHyperVVMSetting -Name '{40E0AC32-46A5-438A-A0B2-2B479E8F2E90}' -DefaultInboundAction Allow
    You may need to reboot your laptop to apply changes.

Finally, pull down the source code for the lab:

  • From PowerShell, open a WSL terminal by running wsl
  • Navigate to your home directory by running cd ~
  • Create a folder for your ROS 2 workspace using mkdir -p ros2_ws/src
  • Navigate to the src folder using cd ros2_ws/src
  • Run git clone -b lab1 https://github.com/UNCC-Embedded-Lab/ROMI-ROS2.git

Step 4: Install & Configure VS Code

For this class, it is recommended that you use VS Code (or a fork like Cursor) to develop code. The general workflow will be:

  • Develop code on laptop using VS Code
  • Do what local testing is possible from laptop
  • Deploy code to RPi using sync_to_pi.sh (a shell script that has been written for you)

In order to set up VS Code for our ROS 2 development:

  2. When prompted to “Select Additional Tasks” during installation, be sure to check the "Add to PATH" option so you can easily open a folder in WSL using the code command.

Once this is complete, you can use VS Code for your code development by opening VS Code > Ctrl+Shift+P > search for and click on WSL: Connect to WSL using Distro… > select the Ubuntu distribution that we installed earlier. Step 5: End-to-End Validation

You now have all the tools you will need to get started developing in ROS 2. We will begin by booting up a simple system that will blink some LEDs on the Romi32U4 Board.

Hardware Assembly & Power

  • Seating the Pi: Mount the Raspberry Pi 4B onto the Romi32U4 pin header. It only fits in one orientation.
  • Power: Insert your 6 AA batteries into the Romi chassis (open the compartment on the underside).
  • Booting: The first boot takes a few minutes, so be patient. Once ready, the Pi creates a WiFi AP named “Romi-XXXX” (last 4 digits of Pi MAC address—labeled on Pi).
Warning: USB power is insufficient for motors. Labs will require battery power to prevent the Pi from experiencing brownout resets.

Connecting to the Robot

  • Join the “Romi-XXXX” WiFi network (Password: romi32u4).
  • Open PowerShell and SSH into the Pi via ssh student@192.168.4.1 (Password: romi32u4).

Building the Workspace

ROS 2 uses "workspaces" to organize code. You must build your workspace on both devices to create the necessary environment files.

In WSL Terminal (Laptop):

cd ~/ros2_ws
colcon build --symlink-install
source ~/ros2_ws/install/setup.bash

Deploy to the Pi by navigating to ~/ros2_ws/src/ROMI-ROS2 and syncing your source code using the provided script (Password: romi32u4):

cd ~/ros2_ws/src/ROMI-ROS2
./scripts/sync_to_pi.sh

This script will sync the entire workspace from your WSL instance to your Raspberry Pi using rsync and automatically build the workspace using colcon.

In SSH (Pi): AFTER completing the above step, open an SSH terminal and run:

ros2 pkg prefix romi_base

If you see /home/student/ros2_ws/install/romi_base, you have confirmed that the ROS 2 workspace was synced and built on your Pi successfully.

Priming the ROS 2 Network

Devices only communicate if they share the same Domain ID. In every new terminal window (WSL and SSH), run export ROS_DOMAIN_ID=<x> where 0 ≤ x ≤ 232. Ensure that each terminal is assigned the identical Domain ID.
Note: ROS 2 uses ROS_DOMAIN_ID=0 by default.

Verifying End-to-End Communication

Verify the toolchain by completing and running a distributed setup_test program across your hardware:

With the repository cloned in WSL, navigate to ~/ros2_ws/src/ROMI-ROS2/romi_base/romi_base/setup_test_node.py in VS Code.
Follow the "TODO" comments in the code to complete the Python node, familiarizing yourself with the Romi hardware interface functions and ROS 2 messaging. When your code is complete, sync to your Pi using the ./scripts/sync_to_pi.sh script and test:

On the Pi (SSH): Launch the hardware interface node:

ros2 run romi_base astar_bridge

On the Laptop (WSL): Launch the logic node:

ros2 run romi_base setup_test

Observe the Output

Your WSL terminal will print the robot status, and the physical LEDs on the Romi board should blink while motors spin (YELLOW + forward, GREEN + stop, RED + reverse). When you achieve this operation, you have confirmed your laptop, Pi, and Romi board are successfully communicating via ROS 2 topics and the provided I2C bridge!
