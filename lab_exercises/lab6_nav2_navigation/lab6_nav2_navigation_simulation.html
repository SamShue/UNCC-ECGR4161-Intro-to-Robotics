Objective

In this lab, you will use the Nav2 (ROS2 Navigation) stack to make a simulated Romi robot drive itself autonomously through a virtual environment in Gazebo. You will launch a simulated robot in a pre-built office world, load the matching map, let the robot localize itself, and then simply click a destination. The navigation stack will plan a path, follow it, and avoid obstacles along the way — all without you driving the robot and without any physical hardware.

This is the simulation version of the Nav2 lab. It runs entirely on your laptop, so you do not need the Romi, the Raspberry Pi, the RPLidar, or a physical space to map. Everything — the robot, its LiDAR, the world, and the walls — exists inside the simulator. This lab focuses on running and observing a production-grade autonomous navigation system rather than implementing one; you will not write any navigation code.

Required Materials

  • Laptop running WSL2 with Ubuntu 22.04 and ROS 2 Humble (the same local environment you used in Labs 5 and 6)
  • A working graphical display for WSL (WSLg, which is included with recent Windows 10/11 WSL installs) so the Gazebo and RViz windows can open
  • Several GB of free disk space for the Gazebo packages

No robot required. Unlike the hardware version of this lab, you do not need a completed Lab 5 map, the physical robot, or the same physical room. The simulation ships with its own office world and a matching map, so you can complete this lab from any computer that runs your ROS 2 environment.

Background

Why Simulation?

Roboticists rarely test new navigation software on real hardware first. A simulator lets you run the exact same ROS 2 code against a virtual robot and a virtual world, so you can develop and debug safely, repeatably, and without risking damage to a real robot. Because Nav2 communicates over ROS 2 topics, it cannot tell the difference between a real Romi and a simulated one — it sees the same /scan, /odom, and transforms either way. That is what makes this lab possible: you will run the identical navigation stack you would run on the real robot, just pointed at a simulated one.

Gazebo and the Simulated Romi

This lab uses Gazebo (the Ignition/Gazebo Sim simulator that pairs with ROS 2 Humble). The provided launch files spawn a simulated Romi robot — complete with a working, simulated RPLidar — inside a pre-built office world: a sealed room with a central hallway and several smaller rooms, designed specifically for testing localization and navigation. A software bridge connects the simulator to ROS 2, publishing the simulated LiDAR on /scan, wheel odometry on /odom, the simulation clock on /clock, and the robot’s transforms — exactly the topics Nav2 expects.

Simulated time: Because the world runs in a simulator, every node must use the simulation clock instead of your computer’s wall clock. This is why every command in this lab sets use_sim_time:=true (or uses a parameter file that already does). If some nodes used wall time and others used sim time, their timestamps would not line up and navigation would fail.

What is Nav2?

Nav2 is the standard navigation framework for ROS2. It is not a single program but a collection of coordinated nodes that together turn a goal like “drive to that corner” into safe, real-time wheel commands. Nav2 is used on real commercial robots — warehouse robots, delivery robots, and cleaning robots all run software organized this way.

Nav2 is powerful because it separates navigation into distinct, replaceable pieces. Each piece is a specialized server with one job. The sections below describe each part of the stack you will be running.

The Parts of the Navigation Stack

1. Map Server

The Map Server loads a saved .pgm/.yaml map and publishes it on the /map topic. In this lab it loads the office map that ships with the repository, which matches the simulated office world. This is the fixed, known map of the world — the reference the robot navigates against.

2. AMCL — Localization

AMCL stands for Adaptive Monte Carlo Localization. Its job is to answer “where is the robot inside the known map?” It does this using a particle filter: it maintains a cloud of hundreds of guesses (particles) about the robot’s pose, then compares the live LiDAR scan against the map. Guesses that match the LiDAR well survive; guesses that match poorly are discarded. Over time the particle cloud converges tightly around the robot’s true position.

In simulation, the robot spawns at a known location (the world origin), and the provided localization launch file hands AMCL that starting pose automatically — so the particle cloud starts already clustered around the robot and the robot is localized from the moment it comes up. You will still see the particle cloud in RViz, and you can nudge it with the 2D Pose Estimate tool if it ever drifts.

3. Costmaps — the Global and Local Costmap

A costmap is a version of the map where every cell is assigned a “cost” that represents how dangerous or undesirable it is for the robot to be there. Walls have very high cost, open space has low cost, and cells near walls have a medium inflation cost that keeps the robot from clipping corners. Nav2 uses two costmaps:

  • Global Costmap: covers the entire map. It is built mostly from the static office map and is used for long-range route planning.
  • Local Costmap: a small window that moves with the robot. It is built from the live (simulated) LiDAR scan and is used to react to obstacles that are not in the static map — such as a box you drop into the world while the robot is driving.

4. Planner Server — the Global Planner

The global planner computes a complete route from the robot’s current position all the way to the goal, using the global costmap. It searches the map for the shortest safe path, treating the problem like finding a route through a maze. Nav2’s default global planner (NavFn) uses a Dijkstra/A*-style search over the costmap grid. The result is the long, smooth line you will see drawn from the robot to the goal in RViz.

The global planner is strategic: it plans the whole trip, but it does not directly drive the wheels and it does not react instantly to new obstacles.

5. Controller Server — the Local Planner

The local planner (also called the controller) is what actually drives the robot. It takes the global path and, using the local costmap, generates the real-time velocity commands sent to /cmd_vel. Nav2’s default controller (the DWB controller) works by simulating many short candidate motions a fraction of a second into the future, scoring each one on how well it follows the global path and how well it avoids obstacles, and then executing the best one. It repeats this many times per second.

The local planner is tactical: it follows the global plan closely but continuously adjusts to stay on course and swerve around anything unexpected in the local costmap.

Key distinction: The global planner decides which way to go across the whole map. The local planner decides how to actually move the wheels right now to follow that route and dodge obstacles. Both run at the same time, at different rates and different scopes.

6. Behavior Tree Navigator (BT Navigator)

The BT Navigator is the coordinator that ties everything together. It runs a behavior tree — a flowchart-like structure — that decides what to do and when: ask the global planner for a path, hand it to the controller, monitor progress, and decide what to do if something goes wrong. It is the “brain” that sequences the other servers.

7. Behavior Server — Recovery Behaviors

When the robot gets stuck — for example, boxed in by obstacles or unable to find a valid path — the BT Navigator triggers recovery behaviors from the Behavior Server. Typical recoveries include clearing the costmaps (in case bad sensor data made a phantom obstacle), spinning in place to look around, backing up, and waiting for a dynamic obstacle to move. These behaviors are what let the robot recover on its own instead of simply giving up.

8. Lifecycle Manager

All of these Nav2 nodes are managed (lifecycle) nodes, meaning they must be explicitly configured and activated in the correct order before navigation can start. The Lifecycle Manager handles this startup sequence automatically so all the servers come up cleanly together. When you launch Nav2, watch for a log message indicating that the lifecycle manager has activated the nodes — that is your signal the stack is ready.

How These Parts Work Together

The diagram below shows the flow from your click to the robot’s (simulated) wheels:

You click a goal in RViz
        ↓
BT Navigator  (coordinates everything)
        ↓
Global Planner  →  computes full route using the Global Costmap (static office map)
        ↓
Local Planner   →  follows route + dodges obstacles using the Local Costmap (simulated LiDAR)
        ↓
/cmd_vel  →  wheel velocity commands  →  simulated Romi drives itself in Gazebo

Meanwhile:  AMCL continuously answers "where am I in the map?" using the simulated LiDAR.

Procedure

Before You Begin: Get the Software and Build the Workspace

Everything in this lab runs in your local WSL environment. You will download the robot software, install Gazebo and Nav2, and build the workspace once.

1. Get the latest robot software

Download a fresh copy of the repository into your ROS 2 workspace’s src directory:

git clone https://github.com/UNCC-Embedded-Lab/ROMI-ROS2.git ~/ros2_ws/src/ROMI-ROS2

Already have a copy? If a ~/ros2_ws/src/ROMI-ROS2 folder already exists from a previous lab, delete it first so you get a clean, up-to-date copy, then clone again:

rm -rf ~/ros2_ws/src/ROMI-ROS2
git clone https://github.com/UNCC-Embedded-Lab/ROMI-ROS2.git ~/ros2_ws/src/ROMI-ROS2

2. Install Gazebo and Nav2 (one time only)

You most likely have not installed Gazebo yet. Install it — along with the ROS 2↔Gazebo bridge, Nav2, and the keyboard teleop tool — from prebuilt binaries for ROS 2 Humble. This is a sizable download, so give it a few minutes:

sudo apt update
sudo apt install ros-humble-ros-gz \
    ros-humble-navigation2 \
    ros-humble-nav2-bringup \
    ros-humble-teleop-twist-keyboard

The ros-humble-ros-gz package pulls in Gazebo (the version that pairs with Humble) plus the ros_gz_sim and ros_gz_bridge packages the simulation launch file needs. You only need to run this install once.

Troubleshooting — “libdiagnostic_updater.so: cannot open shared object file”: On some images the Nav2 binaries install without one of their runtime dependencies. If Nav2 fails to launch later with an error mentioning libdiagnostic_updater.so, install the missing package:

sudo apt install ros-humble-diagnostic-updater

3. Build and source the workspace

Build the romi_base package (which contains the simulation launch files, the office world, and the map) in your local workspace, then source it:

cd ~/ros2_ws
source /opt/ros/humble/setup.bash
colcon build --symlink-install
source install/setup.bash

Source every terminal. This lab uses several terminals. In each new terminal you open, source both ROS 2 and your workspace before running any command, or the ros2 commands and the romi_base launch files will not be found:

source /opt/ros/humble/setup.bash
source ~/ros2_ws/install/setup.bash

1. Launch the Simulated Robot and World

In Terminal 1 (sourced as described above), launch the Gazebo simulation. This starts Gazebo, loads the office world, spawns the simulated Romi with its LiDAR, and starts the bridge that publishes /scan, /odom, /clock, and the robot’s transforms into ROS 2:

ros2 launch romi_base romi_gz.launch.py

A Gazebo window should open showing the office environment with the Romi sitting in it. Press the play button (▶) in the lower-left of the Gazebo window if the simulation is paused, so time starts advancing.

Side Note — Gazebo won’t render? Gazebo is even more sensitive than RViz to WSL’s limited GPU support. This lab’s launch file already forces software rendering for you, but if Gazebo still shows a black screen, freezes, or fails to open, stop it with Ctrl+C and set these variables in the terminal before launching again so both the OpenGL layer and Gazebo’s Qt interface fall back to software:

export LIBGL_ALWAYS_SOFTWARE=1
export QT_QUICK_BACKEND=software
ros2 launch romi_base romi_gz.launch.py

These only affect the terminal you set them in. To make them permanent, add both export lines to the end of your ~/.bashrc.

Gazebo can be slow in WSL. The launch file forces software rendering so it works without GPU passthrough, which means the simulator may run slowly and take a while to open the first time. Give it a minute. If the window is black or empty at first, wait for the world to finish loading. You can verify the bridge is working from another sourced terminal with ros2 topic echo /scan --once — you should see LiDAR ranges.

2. Launch Localization (Map Server + AMCL)

In Terminal 2, launch the simulation localization stack. This starts the Map Server (loading the bundled office map), AMCL, and a lifecycle manager that configures and activates both automatically. It also hands AMCL the robot’s known starting pose at the world origin, so the robot localizes itself immediately:

ros2 launch romi_base romi_amcl_sim.launch.py

Watch the log output. Within a few seconds the lifecycle manager should report that it has activated the localization nodes (the Map Server and AMCL). The costmaps, planners, and controller are not running yet — you will start them in Step 5.

Note: This launch file is sim-only. It loads the romi_office_map.yaml that matches the Gazebo office world and uses a parameter file with use_sim_time: true and a fixed initial pose — you do not need to pass any arguments. Make sure the simulation from Step 1 is running and unpaused first, so AMCL has live /scan and /odom data.

3. Launch the Nav2 RViz View

In Terminal 3, launch the preconfigured Nav2 RViz view. This RViz configuration already has the navigation displays and the goal tools set up for you:

ros2 launch nav2_bringup rviz_launch.py

Side Note — RViz won’t render? In WSL, RViz sometimes opens as a black or blank window, fails to draw, or crashes on startup because of OpenGL/GPU limitations. If that happens, force RViz to use software rendering by setting this environment variable in the same terminal before launching it:

export LIBGL_ALWAYS_SOFTWARE=1
ros2 launch nav2_bringup rviz_launch.py

The variable only affects the terminal you set it in, so run it in each terminal where you start RViz. To make it permanent, add export LIBGL_ALWAYS_SOFTWARE=1 to the end of your ~/.bashrc.

You should see the office map appear as a black-and-white occupancy grid, with the robot and its LiDAR scan on top of it. If the map does not appear, confirm that the fixed frame at the top of the Displays panel is set to map and that localization launched without error.

Take a moment to locate the Nav2 Goal and 2D Pose Estimate tools in the RViz toolbar at the top of the window — you will use them shortly.

4. Confirm the Robot is Localized

Because the robot spawned at the world origin and the localization launch handed AMCL that pose, the robot should already be localized. Confirm it in RViz:

  • The AMCL particle cloud (a PoseArray of small arrows) should appear as a tight cluster around the robot.
  • The colored LiDAR scan points should lie on top of the black walls of the map.

If the scan does not line up with the walls (for example after you have driven the robot around manually), use the 2D Pose Estimate tool: click it, then click on the map where the robot actually is and drag in the direction it is facing before releasing. This re-seeds the particle cloud. Good navigation is impossible if localization is wrong.

5. Launch the Navigation Servers

Now that the robot is localized, bring up the rest of Nav2 — the costmaps, the global planner, the local planner (controller), the BT Navigator, and the Behavior Server. In Terminal 4, pass the Romi Nav2 parameter file and enable simulated time:

ros2 launch nav2_bringup navigation_launch.py \
    use_sim_time:=true \
    params_file:=$HOME/ros2_ws/src/ROMI-ROS2/romi_base/config/nav2_params.yaml

Why a custom parameter file? The repository ships romi_base/config/nav2_params.yaml, a Nav2 configuration tuned for the Romi. Nav2’s stock defaults assume a much larger TurtleBot3-class robot, so they over-inflate obstacles and make narrow corridors look blocked. This file sets the robot’s real footprint (radius ~0.11 m), a tighter inflation radius, and slower velocity limits so the small Romi can plan and drive through the tight office corridors.

Again watch for the lifecycle manager to report that it has activated the navigation nodes. With localization already set and these servers up, the full stack is ready to accept goals.

6. Send a Navigation Goal

Now let the robot drive itself:

  1. In RViz, pick a clear, reachable spot in the office — for example a different room or the far end of the hallway.
  2. Click the Nav2 Goal tool in the RViz toolbar.
  3. Click a destination on the map and drag to set the final heading, then release.

As soon as you release, Nav2 goes to work. You should observe, in order:

  • A global path (a colored line) is drawn from the robot to the goal — this is the global planner output.
  • The simulated robot begins driving on its own in Gazebo, and a shorter path or trajectory near the robot updates continuously — this is the local planner / controller at work.
  • The robot follows the route and arrives at the goal, then stops.

You did not drive the robot — you only chose a destination. The stack handled planning, control, and localization automatically. Watch both windows: the robot moving in Gazebo and its estimated pose tracking along in RViz.

7. Observe the Stack in Action

Send the robot to several different goals around the office and watch how the stack behaves. To clearly see each component, toggle these displays in the RViz Displays panel:

  • Global Costmap — the inflated map used for planning. Notice the “buffer” of cost around every wall.
  • Local Costmap — the small window around the robot built from the simulated LiDAR.
  • Global Plan / Path — the full route from the global planner.
  • Local Plan / Trajectory — the short-term path the controller is executing.
  • AMCL Particle Cloud (PoseArray) — the localization particles.

8. Test Dynamic Obstacle Avoidance

The office map is static, but you can add a brand-new obstacle at runtime and watch Nav2 react to it — just like dropping a box in front of the real robot. Send the robot toward a goal on the far side of the room, then place an obstacle in its path inside Gazebo:

  1. In the Gazebo window, use the shape buttons in the top toolbar (box, sphere, or cylinder) to grab a simple shape.
  2. Click to drop the shape onto the floor in the corridor ahead of the moving robot, between the robot and its goal.

Watch what happens:

  • The new shape appears in the local costmap as the simulated LiDAR detects it.
  • The local planner steers around it while still heading toward the goal.
  • If the robot is fully blocked, a recovery behavior may trigger — you may see it clear its costmaps, spin in place, or back up before trying again.

Tip: Place the obstacle in a spot where the robot still has room to go around it. If you completely wall off a narrow corridor, the robot cannot get through and will (correctly) run its recovery behaviors and eventually report that the goal is unreachable — which is also worth observing.

Optional — drive the robot yourself: To see localization work harder, you can drive the simulated robot manually with the keyboard in a separate sourced terminal (ros2 run teleop_twist_keyboard teleop_twist_keyboard) and watch the particle cloud tighten as it moves. If you do this, close the teleop terminal before sending Nav2 goals, since teleop and Nav2 both publish to /cmd_vel and will fight for control.
