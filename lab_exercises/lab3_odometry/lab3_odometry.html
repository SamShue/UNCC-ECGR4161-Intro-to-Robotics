Background

What is Odometry?

Odometry is the use of motion sensor data to estimate a robot’s change in position over time. For the Romi, we use the wheel encoders to estimate how far each wheel has traveled. By combining these individual wheel distances with the robot’s differential drive kinematics, we can estimate the robot’s overall pose, which includes position and heading relative to its starting location.

Note: Odometry estimates motion by repeatedly adding small changes in position. This means small errors can accumulate over time. Wheel slip, uneven floors, inaccurate wheel diameter, and small wheelbase errors can all cause odometry drift.

Forward Kinematics and Path Integration

In Lab 2, you used inverse kinematics to convert a desired robot velocity into left and right wheel velocities. In this lab, we do the reverse: we use measured left and right wheel motion to estimate how the robot moved.

To calculate the robot’s new position, we translate the wheel distances (dL and dR) into a change in the robot’s pose:

  • Linear displacement: the average distance traveled by the center of the robot.
    Δs = (dL + dR) / 2
  • Angular displacement: the change in the robot’s heading.
    Δθ = (dR - dL) / L
    where L is the wheelbase distance between the two wheels.

Using these displacements, we update the robot’s pose at every time step:

xnew = xold + Δs · cos(θold)
ynew = yold + Δs · sin(θold)
θnew = θold + Δθ

Why Use the Square Test?

In Lab 2, you created a square test node that commands the robot to drive in a 2 ft x 2 ft square using Twist messages. In this lab, you will use that same square motion to verify your odometry implementation.

This is useful because the robot should return close to its starting position after completing the square. If your odometry is working correctly, RViz should show an estimated path that looks like a square, and the final estimated pose should be close to the starting pose.

4. Launch the Robot Stack

In the SSH terminal connected to the Raspberry Pi, launch the Romi core stack:

ros2 launch romi_base romi_core.launch.py

This launch file should start the hardware bridge and your base controller node.

5. Launch RViz

Open a local WSL terminal on your laptop and launch RViz:

rviz2

Side Note — RViz won’t render? In WSL, RViz sometimes opens as a black or blank window, fails to draw, or crashes on startup because of OpenGL/GPU limitations. If that happens, force RViz to use software rendering by setting this environment variable in the same terminal before launching it:

export LIBGL_ALWAYS_SOFTWARE=1
rviz2

The variable only affects the terminal you set it in, so run it in each terminal where you start RViz. To make it permanent, add export LIBGL_ALWAYS_SOFTWARE=1 to the end of your ~/.bashrc.

Configure RViz as follows:

  • In the upper-left panel, set the Fixed Frame to odom.
  • Click Add.
  • Select the By Topic tab.
  • Add the /odom topic.
  • Click Add again.
  • Add a TF display.

6. Run the Square Test Node

Instead of manually driving a circular path or an L-shaped path, this lab uses the automated square test node from Lab 2 as the odometry verification motion.

Place the robot on the floor and clearly mark its starting position. Make sure there is enough open space for the robot to drive a 2 ft x 2 ft square.

In a terminal where the workspace has been sourced, run:

ros2 run romi_base square_test_node

The robot should drive forward, turn 90 degrees, and repeat until it has attempted a complete square. As the robot moves, RViz should update the robot’s estimated path using your odometry calculations.

Important: Do not modify the square test timing values to make the odometry appear better. The square test is being used as a consistent verification motion. If the physical robot does not track the square well, tune the Lab 2 velocity controller. If RViz does not show a reasonable square-shaped path, debug your Lab 3 odometry implementation.
