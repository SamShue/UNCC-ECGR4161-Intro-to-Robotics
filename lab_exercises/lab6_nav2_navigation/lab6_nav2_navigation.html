Objective

In this lab, you will use the Nav2 (ROS2 Navigation) stack to make your Romi robot drive itself autonomously through the environment you mapped in Lab 5. You will load your saved occupancy grid map, give the robot a rough starting position with a single click, and then simply click a destination. The navigation stack will plan a path, follow it, and avoid obstacles along the way — all without you driving the robot.

This lab focuses on running and observing a production-grade autonomous navigation system rather than implementing one. You will not write any navigation code. Your job is to launch the stack correctly, understand what each part of it does, and command the robot to reach goals on its own.

Background

From Mapping to Navigation

In Lab 5 you used SLAM to answer two questions at once: where am I and what does the environment look like. Now that you already have a finished map, the problem changes. The map is fixed and known, so the robot no longer needs to build it. Instead it needs to:

  • Localize: figure out where it is within the existing map.
  • Plan: compute a route from its current position to a goal you choose.
  • Control: follow that route while reacting to obstacles that were not part of the original map.

This is called autonomous navigation, and it is exactly what the Nav2 stack is built to do.

What is Nav2?

Nav2 is the standard navigation framework for ROS2. It is not a single program but a collection of coordinated nodes that together turn a goal like “drive to that corner” into safe, real-time wheel commands. Nav2 is used on real commercial robots — warehouse robots, delivery robots, and cleaning robots all run software organized this way.

Nav2 is powerful because it separates navigation into distinct, replaceable pieces. Each piece is a specialized server with one job. The sections below describe each part of the stack you will be running.

The Parts of the Navigation Stack

1. Map Server

The Map Server loads your saved .pgm/.yaml map from Lab 5 and publishes it on the /map topic. This is the fixed, known map of the world. Unlike Lab 5, this map does not change as the robot drives — it is the reference the robot navigates against.

2. AMCL — Localization

AMCL stands for Adaptive Monte Carlo Localization. Its job is to answer “where is the robot inside the known map?” It does this using a particle filter: it maintains a cloud of hundreds of guesses (particles) about the robot’s pose, then compares the live LiDAR scan against the map. Guesses that match the LiDAR well survive; guesses that match poorly are discarded. Over time the particle cloud converges tightly around the robot’s true position.

AMCL can even start with no idea where the robot is and spread its particles across the whole map until they converge, but that requires driving the robot around so it can gather enough LiDAR to rule out wrong guesses. To save time, in this lab you will instead give AMCL a rough starting pose with a single click using the 2D Pose Estimate tool in RViz. That seeds the particle cloud near the robot’s true location, and AMCL then refines it from the live LiDAR — no driving needed to localize.

3. Costmaps — the Global and Local Costmap

A costmap is a version of the map where every cell is assigned a “cost” that represents how dangerous or undesirable it is for the robot to be there. Walls have very high cost, open space has low cost, and cells near walls have a medium inflation cost that keeps the robot from clipping corners. Nav2 uses two costmaps:

  • Global Costmap: covers the entire map. It is built mostly from your static Lab 5 map and is used for long-range route planning.
  • Local Costmap: a small window that moves with the robot. It is built from the live LiDAR scan and is used to react to obstacles that are not in the static map — a backpack on the floor, a person’s legs, a chair that was moved.

4. Planner Server — the Global Planner

The global planner computes a complete route from the robot’s current position all the way to the goal, using the global costmap. It searches the map for the shortest safe path, treating the problem like finding a route through a maze. Nav2’s default global planner (NavFn) uses a Dijkstra/A*-style search over the costmap grid. The result is the long, smooth line you will see drawn from the robot to the goal in RViz.

The global planner is strategic: it plans the whole trip, but it does not directly drive the wheels and it does not react instantly to new obstacles.

5. Controller Server — the Local Planner

The local planner (also called the controller) is what actually drives the robot. It takes the global path and, using the local costmap, generates the real-time velocity commands sent to /cmd_vel. Nav2’s default controller (the DWB controller) works by simulating many short candidate motions a fraction of a second into the future, scoring each one on how well it follows the global path and how well it avoids obstacles, and then executing the best one. It repeats this many times per second.

The local planner is tactical: it follows the global plan closely but continuously adjusts to stay on course and swerve around anything unexpected in the local costmap.

Key distinction: The global planner decides which way to go across the whole map. The local planner decides how to actually move the wheels right now to follow that route and dodge obstacles. Both run at the same time, at different rates and different scopes.

6. Behavior Tree Navigator (BT Navigator)

The BT Navigator is the coordinator that ties everything together. It runs a behavior tree — a flowchart-like structure — that decides what to do and when: ask the global planner for a path, hand it to the controller, monitor progress, and decide what to do if something goes wrong. It is the “brain” that sequences the other servers.

7. Behavior Server — Recovery Behaviors

When the robot gets stuck — for example, boxed in by obstacles or unable to find a valid path — the BT Navigator triggers recovery behaviors from the Behavior Server. Typical recoveries include clearing the costmaps (in case bad sensor data made a phantom obstacle), spinning in place to look around, backing up, and waiting for a dynamic obstacle to move. These behaviors are what let the robot recover on its own instead of simply giving up.

8. Lifecycle Manager

All of these Nav2 nodes are managed (lifecycle) nodes, meaning they must be explicitly configured and activated in the correct order before navigation can start. The Lifecycle Manager handles this startup sequence automatically so all the servers come up cleanly together. When you launch Nav2, watch for a log message indicating that the lifecycle manager has activated the nodes — that is your signal the stack is ready.

How These Parts Work Together

The diagram below shows the flow from your click to the robot’s wheels:

You click a goal in RViz
        ↓
BT Navigator  (coordinates everything)
        ↓
Global Planner  →  computes full route using the Global Costmap (static Lab 5 map)
        ↓
Local Planner   →  follows route + dodges obstacles using the Local Costmap (live LiDAR)
        ↓
/cmd_vel  →  wheel velocity commands  →  Romi drives itself

Meanwhile:  AMCL continuously answers "where am I in the map?" using the LiDAR.

Procedure

Before You Begin: Get the Latest Robot Software

The robot software has been updated for this lab. It now includes a clock-sync script and a Nav2 parameter file tuned for the Romi, both of which you will use in the steps below. Before you start, download a fresh copy of the repository to your local WSL environment so you have the latest versions of these files. In a local WSL terminal, clone the repository into your ROS 2 workspace’s src directory:

git clone https://github.com/UNCC-Embedded-Lab/ROMI-ROS2.git ~/ros2_ws/src/ROMI-ROS2

This downloads the current version from the repository’s main branch. You are only grabbing a fresh copy of the files — you do not need to maintain or update this folder afterward.

Already have a copy? If a ~/ros2_ws/src/ROMI-ROS2 folder already exists from a previous attempt, delete it first so you get a clean, up-to-date copy, then clone again:

rm -rf ~/ros2_ws/src/ROMI-ROS2
git clone https://github.com/UNCC-Embedded-Lab/ROMI-ROS2.git ~/ros2_ws/src/ROMI-ROS2

Everything this lab needs from the repository — the scripts/sync_pi_clock.sh script and the romi_base/config/nav2_params.yaml parameter file — lives in this ~/ros2_ws/src/ROMI-ROS2 folder. The commands below assume that path.

1. Prepare the Robot and Connect

  1. Insert charged batteries into the Romi and turn it on.
  2. Connect your laptop to the Romi’s Wi-Fi network. As in Lab 5, ROS2 traffic flows over this connection, so the robot stack running on the Pi and the software running locally on your laptop will automatically discover each other.
  3. Place the robot in the same environment you mapped in Lab 5, ideally near a distinctive feature like a corner or a doorway. Starting near recognizable geometry helps the robot localize itself more quickly.

2. Sync the Raspberry Pi Clock, Then Launch the Romi Robot Stack

The Raspberry Pi runs on the robot’s isolated Wi-Fi with no internet access and no battery-backed clock, so its wall clock drifts every time it boots. Nav2 and AMCL rely on message timestamps and transforms being closely aligned across the Pi and your laptop — a skewed Pi clock causes “message filter dropping message” and “transform data too old” errors that make localization and navigation misbehave. Before launching the robot stack, you will run the provided sync script from your local WSL environment to push your laptop’s time onto the Pi over SSH.

One-time setup (run once per Pi): The very first time you use this script, run it once with the --install-sudoers flag. Setting the clock requires sudo on the Pi, and the delay of typing the sudo password skews the very clock offset you are trying to correct. This one-time step installs a small, clock-command-scoped passwordless-sudo rule on the Pi so later syncs run instantly and accurately:

cd ~/ros2_ws/src/ROMI-ROS2
./scripts/sync_pi_clock.sh --install-sudoers

You only need to do this once per Pi. During this one-time setup you may be prompted for the SSH and sudo password.

After that one-time setup, sync the clock by running the script without any flags:

cd ~/ros2_ws/src/ROMI-ROS2
./scripts/sync_pi_clock.sh

The script SSHes into the Pi (default user student, host 192.168.4.1), sets its clock to match your laptop, and verifies the offset is small. Because you installed the passwordless-sudo rule above, it no longer prompts for the sudo password (you may still be asked for the SSH password).

Do this every time. Run sync_pi_clock.sh before launching the robot stack every session (and again if the Pi has been powered on for a long time), so the control loop and Nav2 start with a corrected clock. This uses the copy of the repository you cloned in Before You Begin.

Once the clock is synced, SSH into the Raspberry Pi in Terminal 1 and launch the Romi robot stack. As in Lab 5, this starts both the Romi base controller and the RPLidar driver, providing the /scan and /odom topics and the robot’s transforms:

ssh student@192.168.4.1
ros2 launch romi_base romi_robot.launch.py

Confirm the terminal is running without error and the LiDAR is spinning before continuing.

3. Make Sure Your Map is Available

Nav2 needs to read the map you saved in Lab 5. It should already be in your local WSL home directory as ~/map.yaml and ~/map.pgm. Confirm both files are present:

ls -lh ~/map.yaml ~/map.pgm

If you renamed your files for the Lab 5 submission (for example lastname_firstname_lab5.yaml), either copy them back to ~/map.yaml and ~/map.pgm, or simply use your actual filename in the launch command in the next step. The .yaml file’s image: line must point to the matching .pgm file, and both must be in the same folder.

4. Install Nav2 (One Time Only)

If you have not already installed Nav2 in your local WSL environment, install it now. Like slam_toolbox, it is available as a prebuilt binary for ROS2 Humble:

sudo apt update
sudo apt install ros-humble-navigation2 ros-humble-nav2-bringup

You only need to do this once.

Troubleshooting — “libdiagnostic_updater.so: cannot open shared object file”: On some images the Nav2 binaries install without pulling in one of their runtime dependencies, the diagnostic_updater library. If Nav2 fails to launch in Step 5 with an error like this:

[nav2_container]: Failed to load library: Could not load library dlopen error:
libdiagnostic_updater.so: cannot open shared object file: No such file or directory

install the missing package, then launch Nav2 again:

sudo apt update
sudo apt install ros-humble-diagnostic-updater

If a launch is still running, stop it with Ctrl+C first, then re-run the Nav2 launch command from Step 5. This is also a one-time fix.

5. Launch Localization (Map Server + AMCL)

Instead of starting everything at once, you will bring up navigation in two stages: localization first — so the robot can figure out where it is — and then the rest of the stack. In a local WSL terminal (Terminal 2), launch just the Map Server and AMCL, pointed at your saved map and the Romi Nav2 parameter file:

ros2 launch nav2_bringup localization_launch.py \
    use_sim_time:=false \
    map:=$HOME/map.yaml \
    params_file:=$HOME/ros2_ws/src/ROMI-ROS2/romi_base/config/nav2_params.yaml

Why a custom parameter file? The repository now ships romi_base/config/nav2_params.yaml, a Nav2 configuration tuned for the Romi. Nav2’s stock defaults assume a much larger TurtleBot3-class robot, so they over-inflate obstacles and make narrow corridors look blocked. This file sets the robot’s real footprint (radius ~0.11 m), a tighter inflation radius, and slower velocity limits so the small Romi can plan and drive through tight spaces. You pass the same file to both the localization launch here and the navigation launch later.

Watch the log output. After a few seconds the Lifecycle Manager should report that it has activated the localization nodes (the Map Server and AMCL). The costmaps, planners, and controller are not running yet — you will start them once the robot is localized.

Important: The Romi stack from Step 2 must be running and publishing /scan and /odom before you launch localization. AMCL needs live LiDAR and odometry to match the robot against the map. If the robot stack is not running, AMCL will start but the robot will never be able to localize.

6. Launch the Nav2 RViz View

In a local WSL terminal (Terminal 3), launch the preconfigured Nav2 RViz view. This RViz configuration already has all the navigation displays and the special goal tools set up for you:

ros2 launch nav2_bringup rviz_launch.py

Side Note — RViz won’t render? In WSL, RViz sometimes opens as a black or blank window, fails to draw, or crashes on startup because of OpenGL/GPU limitations. If that happens, force RViz to use software rendering by setting this environment variable in the same terminal before launching it:

export LIBGL_ALWAYS_SOFTWARE=1
ros2 launch nav2_bringup rviz_launch.py

The variable only affects the terminal you set it in, so run it in each terminal where you start RViz. To make it permanent, add export LIBGL_ALWAYS_SOFTWARE=1 to the end of your ~/.bashrc.

You should see your Lab 5 map appear as a black-and-white occupancy grid. If the map does not appear, confirm that the fixed frame at the top of the Displays panel is set to map and that localization launched without error.

Take a moment to locate these tools in the RViz toolbar at the top of the window:

  • Nav2 Goal — used to command the robot to drive to a destination (you will use this once the robot is localized and the navigation servers are running).
  • 2D Pose Estimate — you will use this to tell AMCL roughly where the robot is in the map (Step 7).
  • Publish Point — not needed for this lab.

7. Set the Robot’s Initial Pose

AMCL starts out with no idea where the robot is. Rather than driving the robot around to let it localize itself, you will simply tell it roughly where it is using the 2D Pose Estimate tool. This is much faster and is all AMCL needs to lock on.

  1. Click the 2D Pose Estimate tool in the RViz toolbar.
  2. Click on the map at the spot where the robot actually is, and — while holding the mouse button — drag in the direction the robot is physically facing, then release. The click sets position; the drag sets heading.

As soon as you release, the AMCL particle cloud (a PoseArray of small arrows) appears as a tight cluster around the pose you gave, and the live LiDAR scan should snap onto the map’s walls.

Check your localization: The red/colored LiDAR points should lie on top of the black walls of the map. If the scan is clearly rotated or offset from the walls, your initial estimate was off — just use the 2D Pose Estimate tool again and click/drag a more accurate pose. Take a moment to get this right: good navigation is impossible if localization is wrong.

8. Launch the Navigation Servers

Now that the robot knows where it is, bring up the rest of Nav2 — the costmaps, the global planner, the local planner (controller), the BT Navigator, and the Behavior Server. In a local WSL terminal (Terminal 4), using the same Romi parameter file:

ros2 launch nav2_bringup navigation_launch.py \
    use_sim_time:=false \
    params_file:=$HOME/ros2_ws/src/ROMI-ROS2/romi_base/config/nav2_params.yaml

Again watch for the Lifecycle Manager to report that it has activated the navigation nodes. With localization already set and these servers up, the full stack is ready to accept goals. AMCL keeps refining the robot’s pose the whole time it drives.

9. Send a Navigation Goal

Now let the robot drive itself:

  1. Make sure there is a clear, reachable spot in the environment you want the robot to drive to.
  2. Click the Nav2 Goal tool in the RViz toolbar.
  3. Click a destination on the map and drag to set the final heading, then release.

As soon as you release, Nav2 goes to work. You should observe, in order:

  • A global path (a colored line) is drawn from the robot to the goal — this is the global planner output.
  • The robot begins driving on its own, and a shorter path or trajectory near the robot updates continuously — this is the local planner / controller at work.
  • The robot follows the route and arrives at the goal, then stops.

You did not drive the robot — you only chose a destination. The stack handled planning, control, and localization automatically.

10. Observe the Stack in Action

Send the robot to several different goals around the environment and watch how the stack behaves. To clearly see each component, it helps to toggle these displays in the RViz Displays panel:

  • Global Costmap — the inflated map used for planning. Notice the “buffer” of cost around every wall.
  • Local Costmap — the small window around the robot built from live LiDAR.
  • Global Plan / Path — the full route from the global planner.
  • Local Plan / Trajectory — the short-term path the controller is executing.
  • AMCL Particle Cloud (PoseArray) — the localization particles.

11. Test Dynamic Obstacle Avoidance

Send the robot toward a goal, then place a new obstacle in its path that was not part of the Lab 5 map — a shoe, a box, or your foot (carefully). Watch what happens:

  • The obstacle appears in the local costmap as the LiDAR detects it.
  • The local planner steers around it while still heading toward the goal.
  • If the robot is fully blocked, a recovery behavior may trigger — you may see it clear its costmaps, spin in place, or back up before trying again.

Safety: The robot is now moving under its own control. Keep the area clear of feet and cables, and stay ready to catch or power off the robot if it heads somewhere unsafe. Nav2 will not drive off a table edge on its own awareness — the LiDAR only sees walls, not drop-offs.
