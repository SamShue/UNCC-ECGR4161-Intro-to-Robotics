Objective

In this lab, you will use a pre-built SLAM algorithm to generate a 2D occupancy grid map of a real environment with your Romi robot. You will drive the robot through the space manually while the SLAM system processes LiDAR data in real time to build and refine the map. At the end of the lab, you will save the completed map to disk for use in future work.

This lab focuses on using an existing SLAM implementation rather than implementing one yourself. Your primary responsibilities are configuring the software stack, driving the robot effectively, and producing a clean, complete map of the environment.

Background

What is SLAM?

SLAM stands for Simultaneous Localization and Mapping. It is a core capability in mobile robotics that allows a robot to answer two questions at the same time:

  • Where am I? (localization)
  • What does the environment look like? (mapping)

These two problems are intertwined. To build an accurate map, the robot needs to know where it is. But to know where it is, the robot needs a map to compare against. SLAM algorithms solve both problems simultaneously by continuously updating the map and the robot’s estimated position together as new sensor data arrives.

Note: SLAM is an active area of robotics research. This lab uses a well-established, production-quality SLAM implementation so you can focus on the practical aspects of running and using SLAM rather than the underlying mathematics.

The Occupancy Grid Map

The type of map produced by this lab is called an occupancy grid. The environment is divided into a uniform grid of cells, and each cell is assigned one of three states:

  • Occupied (black): A wall or obstacle was detected at this cell.
  • Free (white): The LiDAR measured through this cell, meaning it is open space.
  • Unknown (gray): The robot has not yet gathered enough data to classify this cell.

As the robot explores the environment, more and more cells transition from unknown to either occupied or free. A complete map will have very few gray areas remaining.

The slam_toolbox Package

This lab uses slam_toolbox, a widely used open-source ROS2 SLAM package. It subscribes to the LiDAR scan topic and the robot’s odometry topic, and publishes a continuously updated occupancy grid map on the /map topic. It also broadcasts the transform between the map frame and the odom frame, which connects the robot’s localization estimate to the map coordinate system.

The package will be run in online asynchronous mode. In this mode, slam_toolbox processes incoming sensor data as fast as it can without holding up the rest of the robot system. This is the recommended mode for running SLAM on a live robot.

Driving Strategy for Good Map Quality

Because the robot is manually driven during this lab, the quality of the resulting map depends on how you drive. Keep the following in mind:

  • Drive slowly and smoothly. Fast or jerky motion reduces the quality of the LiDAR scan data and makes it harder for SLAM to track the robot’s position accurately.
  • Expose all walls. Drive the robot close enough to each wall in the environment so the LiDAR can clearly measure it. Walls that are only partially seen will appear incomplete.
  • Revisit areas. Driving the robot back through areas it has already mapped helps slam_toolbox correct small errors and close loops. This is called loop closure.
  • Avoid covering the LiDAR. Your hands, cables, or other objects should not block the spinning sensor while the robot is mapping.

How the Map is Saved

Once you are satisfied with the map, you will save it using the map_saver_cli tool from the nav2_map_server package. This tool reads the current occupancy grid from the /map topic and writes two files to disk:

  • .pgm file: A grayscale image of the map. Black pixels are walls, white pixels are free space, and gray pixels are unknown.
  • .yaml file: Metadata that describes the map, including the resolution (meters per pixel), the origin of the map coordinate frame, and the file path to the image.

Both files are required to fully describe a saved map. You will submit both files as part of this lab.

Procedure

1. Prepare the Robot and Connect

  1. Insert charged batteries into the Romi and turn it on.
  2. Connect your laptop to the Romi’s Wi-Fi network. All ROS2 traffic will flow over this connection, so the robot stack running on the Pi and the software running locally on your laptop will automatically discover each other.
  3. Open terminal windows as needed. You will use SSH to connect to the Raspberry Pi for robot-side software only. slam_toolbox, RViz, and teleoperation all run in your local WSL environment.

2. Launch the Romi Robot Stack

In Terminal 1, SSH into the Raspberry Pi and launch the LiDAR driver:

ssh student@192.168.4.1
ros2 launch rplidar_ros rplidar_a1_launch.py

In Terminal 2, open a new SSH session and launch the Romi base controller and lidar:

ssh student@192.168.4.1
ros2 launch romi_base romi_robot.launch.py

Confirm that both terminals are running without error before continuing. The LiDAR should be spinning visibly on top of the robot.

3. Launch slam_toolbox

In a local WSL terminal, launch slam_toolbox in online asynchronous mode. This runs on your laptop and subscribes to the /scan and /odom topics published by the robot over the Wi-Fi network:

ros2 launch slam_toolbox online_async_launch.py

After a moment, slam_toolbox will begin publishing a map. You will not see any visual output in this terminal, but you should see log messages indicating that the node has started and is processing data.

Important: slam_toolbox requires both the /scan topic and the /odom topic to be actively publishing before it can operate. Make sure the LiDAR driver and the Romi core stack are fully running before launching slam_toolbox.

4. Configure RViz

In Terminal 4 on your local WSL environment, launch RViz:

rviz2

Configure RViz to visualize the mapping process:

  1. In the upper-left panel, set the Fixed Frame to map.
  2. Click Add → By Topic → select /map → click OK.
  3. Click Add → By Topic → select /scan → click OK.
  4. Click Add → By Display Type → select TF → click OK.

If slam_toolbox is running correctly, a small gray square should appear in RViz, representing the initial unknown map centered at the robot’s starting position. The LiDAR scan lines should also be visible, radiating outward from the robot.

5. Drive the Robot to Build the Map

Open a fifth terminal in your local WSL environment and launch the keyboard teleop node:

ros2 run teleop_twist_keyboard teleop_twist_keyboard

Use the keyboard to drive the robot through the environment. Follow the driving strategy described in the Background section. Keep an eye on RViz as you drive — you should see the map growing in real time as the robot explores new areas.

Safety: Keep the teleop terminal in focus while driving. The robot will continue moving at its last commanded velocity if focus is lost. If the robot behaves unexpectedly, press S in the teleop terminal to send a zero-velocity command and stop the robot.

Continue driving until the map in RViz covers the full extent of the environment with minimal gray (unknown) areas remaining. A complete map will clearly show the room’s walls as solid black lines enclosing a white interior.

6. Save the Map

Once you are satisfied with the map, stop driving the robot and leave all nodes running. In a local WSL terminal, save the map using the map saver tool. Because slam_toolbox is running locally, the map is saved directly to your WSL home directory:

ros2 run nav2_map_server map_saver_cli -f ~/map

This will create two files in the home directory of your WSL environment:

  • ~/map.pgm — the map image
  • ~/map.yaml — the map metadata

Verify the files were created:

ls -lh ~/map.*

Both files should be present and non-empty. The .pgm file will typically be several kilobytes in size. If it is 0 bytes, the map was not saved correctly and you should retry the save command while slam_toolbox is still running.

Open the .pgm file with an image viewer to visually confirm it looks like a reasonable map of the room before moving on to submission.
